# 开源量化项目调研与可借鉴知识

本文记录对外部开源量化项目的阶段性调研结论，并结合 AlphaFlow 当前架构整理可借鉴的设计方向。本文属于知识总结，不代表相关能力已经立项或实现；涉及业务规则、风控阈值和系统边界的改动仍需单独评审。

## 调研来源与使用边界

本轮参考资料主要是一篇介绍 Qbot、AI Quant Trade、Abu、Trader、VNPy、QUANTAXIS、Qlib 和 yfinance 的知乎文章，以及文章列出的项目公开仓库。

原文偏项目导览，适合发现方向，不适合作为架构决策或收益判断的唯一依据。文中 Star 数、运行年限、收益率等信息具有时效性或缺少可复核依据，因此本文不将其视为已验证事实。后续若要引入具体设计，应继续阅读对应项目的官方文档和源代码，并验证其维护状态、许可证、实现质量和适用边界。

## 项目能力概览

### Qbot

Qbot 强调从市场数据、策略分析、回测、模拟或实盘执行到消息通知的完整闭环，并通过 Web 和移动端降低使用门槛。

可借鉴知识：

- 系统价值不仅来自单个策略，也来自研究、验证、执行和反馈形成的闭环。
- 面向使用者的运行状态、任务进度、异常通知和结果展示，应与底层交易能力同步建设。
- 多市场支持应建立在稳定的统一模型和适配层上，不能简单堆叠不同市场的接口。

需要警惕：大而全的平台容易形成复杂依赖和模糊边界。AlphaFlow 不应为了表面的一站式体验合并现有服务职责。

### AI Quant Trade

该项目更接近量化学习资料、策略样例和技术路线集合，覆盖传统策略、因子、机器学习、深度学习、强化学习和高频交易等方向。

可借鉴知识：

- 研究代码、生产策略和学习样例应明确分层，避免实验代码直接进入实时交易路径。
- 新模型需要统一的数据集、评价标准、样本外验证和部署约束，不能仅以训练期收益判断效果。
- 研究资料可以形成策略目录，但策略是否可上线仍需通过独立验收流程。

### Abu

Abu 强调多市场回测、择时策略、手续费与滑点，以及网格参数搜索。

可借鉴知识：

- 回测必须显式建模手续费、滑点、成交限制和市场规则。
- 参数搜索的结果必须可复现，并保留数据范围、策略参数、成交模型和评价指标。
- 参数选择需要训练区间与验证区间分离，避免只按最终收益选取参数。
- 不同市场的涨跌停、交易时段、合约规则和流动性约束不应被统一成交模型掩盖。

### Trader

该项目体现了简单趋势策略长期运行和持续维护的思路。

可借鉴知识：

- 策略复杂度不等于策略质量；简单、可解释、可监控的策略更容易验证和长期维护。
- 长期运行系统更应关注回撤、稳定性、交易成本和故障恢复，而不是单次高收益。
- 外部项目声称的历史收益只能作为线索，不能替代可复现回测和独立实盘记录。

### VNPy

VNPy 的核心价值在于统一交易接口、事件驱动架构、策略模块、回测工具和前端风险管理。

可借鉴知识：

- Gateway 不应只统一请求方法，还应描述交易所或市场的能力差异。
- 实时交易系统应通过事件流解耦行情、策略、仓位、执行和账户状态。
- 风控应位于订单到达交易所之前，并结合账户、策略、交易对和时间窗口进行限制。
- 策略框架、执行适配器和风险控制应保持独立边界。

### QUANTAXIS

QUANTAXIS 强调从采集、存储、因子、策略到任务调度的全链路，以及通过异步消息和分析数据库支持扩展。

可借鉴知识：

- 原始事实数据、可重算特征、实时状态和任务消息应由不同存储或基础设施承担。
- 数据获取、指标计算、因子研究和策略执行应通过稳定模型连接，而不是相互导入内部实现。
- 分布式或微服务设计应由吞吐量、隔离性和运维需求驱动，不应成为默认目标。

AlphaFlow 当前使用 ClickHouse、Redis 和 NATS JetStream 分别承担历史事实、实时状态和消息传输，方向与上述原则一致。

### Qlib

Qlib 的主要价值是面向机器学习量化研究的数据管理、模型实验、策略验证和工作流组织。

可借鉴知识：

- 数据集、特征、模型、策略和评价结果需要建立可追踪关系。
- 每次实验应记录数据版本、特征版本、代码版本、参数和评价配置。
- 研究阶段必须重视样本外验证、滚动验证和数据泄漏检查。
- 机器学习能力应建立在可复现的数据与实验基础上，而不是先引入复杂模型。

### yfinance

yfinance 展示了以简单接口降低外部金融数据获取成本的价值。

可借鉴知识：

- 数据客户端应隐藏分页、重试、限频和响应解析细节，为上层提供稳定模型。
- 数据来源必须附带质量、更新时间、市场时区和授权边界。

AlphaFlow 当前聚焦合约交易和事件合约，已有交易所原生数据链路，因此暂时没有直接引入 yfinance 的必要。未来若扩展股票基本面研究，可重新评估。

## AlphaFlow 已具备的对应基础

AlphaFlow 当前已有以下基础，不需要因外部项目而重新建设一套框架：

- Go 实时主路径覆盖行情采集、指标、在线策略、仓位、模拟执行和交易所执行。
- NATS JetStream 解耦行情快照、策略决策和内部补偿任务。
- ClickHouse 保存已闭合 K 线、策略事件、回测交易和回测摘要。
- Redis 保存实时状态、恢复缓存和当前 paper 仓位。
- 在线与回测共享 Go 策略代码，并通过 `strategyframe` 统一上下文。
- 回测使用已闭合历史数据并约束多周期时间边界，降低未来函数风险。
- 已有手续费、返佣、滑点入口和基础回测指标。
- 交易所 REST 与 WebSocket 适配、公共市场模型和历史读写能力已经分层。

因此，调研结论应优先用于补齐可靠性、实验管理和可观测性，而不是替换现有技术栈。

## 值得补齐的能力

### P0：账户级实时风控

这是 testnet 和 live 安全验收前最重要的缺口。风控层可考虑覆盖以下类别，但具体规则和阈值必须经过业务确认：

- 单笔最大名义价值。
- 单账户、单策略和单交易对最大敞口。
- 最大杠杆和保证金占用限制。
- 下单、撤单和失败请求的时间窗口频率限制。
- 连续亏损、单日亏损或回撤熔断。
- 行情过期、账户状态异常、私有流失联时禁止新开仓。
- 全局 kill switch 和只减仓模式。
- 风控拒绝事件的持久化、告警和审计。

风控判断应位于订单意图生成之后、交易所请求之前，同时避免将风控规则散落到各交易所适配器中。

### P0：资金和成交参数配置化

保证金、杠杆、手续费、返佣和滑点等参数需要从共用默认值逐步变成可追踪配置。建议考虑以下覆盖层次：

```text
系统默认
  -> 交易所与市场
  -> 账户
  -> 策略
  -> 单次回测或运行覆盖
```

候选参数包括：

- 保证金和杠杆。
- maker/taker 手续费和返佣。
- 滑点模型和流动性约束。
- 资金费率处理。
- 最小下单量、数量步长、价格步长和最小名义价值。
- 仓位和风险预算上限。

配置合并规则、缺省值和覆盖优先级属于业务规则，实施前需要单独设计。

### P1：批量回测与实验管理

第一阶段适合实现简单、可审计的网格参数展开，而不是立即引入遗传算法或复杂优化器。

每次实验至少应记录：

- run ID 和父实验 ID。
- 策略名称、参数和启用的周期。
- 数据集范围、数据质量检查结果和生成时间。
- 特征或指标口径版本。
- 手续费、滑点、资金费和成交规则。
- 代码版本或可定位的构建版本。
- 收益、最大回撤、profit factor、胜率和稳定性指标。

参数评价应支持样本内、样本外和滚动区间拆分，防止把过拟合结果误认为策略优势。

### P1：回测诊断与可视化

AlphaFlow 已经具备基础回测统计，下一步应优先提高诊断能力：

- 资金曲线和回撤曲线。
- K 线上的开仓、加仓、减仓和平仓标记。
- 按交易对、方向、时间段和退出原因拆分收益。
- 手续费、滑点和资金费对收益的影响。
- 参数实验之间的收益与回撤对比。
- 数据缺口、降级输入和策略错误的时间定位。

图表应基于已有回测事实数据生成，不在前端重复计算关键收益口径。

### P1：数据集与特征协议版本化

为了保证研究和回测可复现，应逐步记录：

- 原始数据来源、时间范围和完整性结果。
- 特征协议版本和指标参数。
- 策略参数与策略代码版本。
- 成交模型和费用配置。
- 研究或回测任务的创建时间、完成状态和错误信息。

版本化的目标是解释结果差异，而不是保存所有可重算中间数据。长期事实仍应以原始 K 线、事件和明确的运行配置为主。

### P2：Gateway 能力模型

统一 Gateway 的重点应是显式描述差异，而不是强行消除差异。候选能力包括：

- 支持的订单类型和 time-in-force。
- reduce-only、单向持仓和双向持仓模式。
- 数量精度、价格精度、最小数量和最小名义价值。
- 杠杆范围和保证金模式。
- 客户端订单 ID、批量订单和撤单能力。
- 私有数据流和 REST 对账能力。
- 限频、可重试错误和不可重试错误分类。

该能力模型可以服务执行前校验和风控，但不应形成覆盖所有交易所细节的庞大抽象层。

### P2：研究层与实时层保持分离

建议继续维持当前总体方向：

- Go 负责实时行情、在线策略、仓位、执行和未来实时风控。
- Python 适合研究、机器学习、实验分析和离线报表。
- ClickHouse 保存长期事实数据。
- Redis 保存实时状态和恢复缓存。
- NATS JetStream 负责服务间事件和内部任务传输。

只有在出现明确的吞吐量、故障隔离或独立部署需求时，才继续拆分服务。

## 暂不建议采用

- 不直接引入 Qlib、VNPy、QUANTAXIS 等完整框架替换现有 Go 主路径。
- 不因项目清单覆盖机器学习、强化学习或大模型，就立即增加对应生产模块。
- 不在缺少统一数据集和实验记录的情况下开展大规模参数搜索。
- 不以外部项目声称的收益率、Star 数或案例数量作为技术选型依据。
- 不为了形成“一站式平台”合并当前边界清晰的服务。
- 不为了统一 Gateway 而隐藏会影响成交和风险的交易所差异。
- 不把可重算指标和特征全部长期落库，避免存储口径失控。

## 建议优先级

建议按以下顺序继续评估，是否实施仍需逐项设计和确认：

1. 账户级实时风控。
2. 资金与成交参数配置化。
3. 批量回测和可复现实验记录。
4. 回测结果查询与图表化诊断。
5. 数据集和特征协议版本化。
6. Gateway 能力模型标准化。
7. 在研究基础成熟后，再评估因子平台和机器学习工作流。
